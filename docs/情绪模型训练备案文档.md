# 情绪模型训练备案文档

## 一、系统概述

### 1.1 系统名称
情绪分析与预测模型系统（Emotion Analysis & Prediction Model System）

### 1.2 系统定位
基于大语言模型的情绪分析系统，用于实时分析用户对话中的情绪状态，提供情绪健康评分、风险预警和改善建议，支持情绪趋势追踪与可视化。

### 1.3 应用场景
- 心理咨询师场景：辅助识别用户情绪状态，提供专业参考
- 日常对话场景：实时监测用户情绪变化，及时预警
- 情绪趋势分析：长期追踪用户情绪变化趋势

## 二、核心模块说明

### 2.1 情绪分析模型

#### 2.1.1 分析结果数据结构

```python
class EmotionAnalysisResult(BaseModel):
    current_mood: str           # 当前情绪状态（如：开心、平静、焦虑、沮丧）
    warning_type: str           # 警告类型（无/焦虑倾向/情绪低落等）
    score: float                # 情绪健康分数 (0-1，越高越好)
    reasons: list[str]          # 分析原因（1-3条）
    related_chats: list[str]    # 相关对话片段（1-3条）
    suggestions: list[str]      # 改善建议（score<0.6时提供）
```

#### 2.1.2 情绪评分标准

| 分数区间 | 情绪等级 | 情绪表现 | 预警级别 |
|----------|----------|----------|----------|
| 0.8 ~ 1.0 | 积极情绪 | 开心、兴奋、满足、感激、自信 | 无 |
| 0.6 ~ 0.8 | 平静情绪 | 平和、正常、无明显波动 | 无 |
| 0.4 ~ 0.6 | 轻微负面 | 轻微烦恼、些许疲惫、小郁闷 | 关注 |
| 0.2 ~ 0.4 | 明显负面 | 焦虑、沮丧、失落、愤怒、难过 | 预警 |
| 0.0 ~ 0.2 | 严重负面 | 极度悲伤、绝望、崩溃、强烈愤怒 | 高危 |

**重要规则**: 积极情绪的score一定大于0.6，负面情绪的score一定小于0.6。

### 2.2 情绪分析流程

```
用户对话输入 → 构建分析Prompt → LLM异步分析 → JSON解析 → 结果存储 → 消息总线广播
```

#### 2.2.1 分析Prompt设计

```
你是一个专业的情绪分析师。请根据用户的对话内容，分析其情绪状态。

分析要求：
1. current_mood：用简短词语描述用户当前的主要情绪
2. warning_type：情绪正常填"无"；存在负面情绪风险填具体类型
3. score：严格按照评分标准打分
4. reasons：简要说明判断依据，1-3条
5. related_chats：引用对话中反映情绪的具体语句，1-3条
6. suggestions：score < 0.6时给出1-2条改善建议；score >= 0.6时填空列表
```

#### 2.2.2 异步处理机制

- **非阻塞设计**: 通过 `asyncio.create_task` 创建后台任务
- **主流程不等待**: 情绪分析在后台执行，不影响对话响应速度
- **消息总线广播**: 分析完成后通过message_bus发送结果

### 2.3 情绪趋势分析

#### 2.3.1 情绪趋势数据结构

```python
class EmotionPoint(BaseModel):
    date: str                    # 日期
    score: int                   # 情绪分数
    emotion: str                 # 情绪标签
    trigger_summary: str         # 触发原因简述

class EmotionTrend(BaseModel):
    points: list[EmotionPoint]   # 情绪趋势点
    summary: str                 # 情绪趋势总结
    advice: str                  # 情绪建议
```

#### 2.3.2 关键时刻识别

```python
class KeyMoment(BaseModel):

    summary: str                 # 简述
```

## 四、中间件架构

### 4.1 EmotionSpeculateMiddleware

**继承**: `AgentMiddleware` (LangGraph)

**生命周期钩子**:
- `abefore_model`: 在模型处理前触发情绪分析

**工作流程**:
1. 从state中提取session_id和messages
2. 获取最新用户消息（最多3条）
3. 创建后台异步任务执行情绪分析
4. 立即返回，不阻塞主流程

### 4.2 消息总线通信

**消息格式**:
```json
{
  "type": "emotion",
  "session_id": "会话ID",
  "emotion_analysis": {
    "current_mood": "焦虑",
    "warning_type": "焦虑倾向",
    "score": 0.35,
    "reasons": ["用户表达了对工作的担忧"],
    "related_chats": ["我最近工作压力很大"],
    "suggestions": ["建议适当休息，放松身心"]
  }
}
```

## 七、性能与安全

### 7.1 性能优化

- **异步分析**: 后台任务不阻塞主对话流程
- **响应长度限制**: 超过10000字符的LLM响应直接跳过
- **空响应处理**: 检测并跳过空响应
- **JSON解析容错**: 支持markdown代码块包裹的JSON

### 7.2 数据安全

- **会话隔离**: 通过session_id实现数据隔离
- **JSON存储**: 情绪分析结果以JSON格式存储，便于查询
- **时间戳追踪**: 记录创建和更新时间

### 7.3 容错机制

- **异常捕获**: 全流程异常捕获与日志记录
- **解析失败处理**: JSON解析失败时跳过分析
- **LLM调用失败**: 记录错误日志，不影响主流程

## 八、训练与优化

### 8.1 Prompt优化方向

1. **评分标准细化**: 明确各分数区间的情绪表现
2. **分析维度扩展**: 增加更多情绪分析维度
3. **建议个性化**: 根据用户历史情绪数据提供个性化建议

### 8.2 模型优化方向

1. **专用情绪模型**: 训练专用情绪分析模型，降低LLM调用成本
2. **多模态情绪分析**: 结合语音语调、面部表情等多模态数据
3. **情绪趋势预测**: 基于历史数据预测未来情绪变化趋势

### 8.3 数据标注规范

- **情绪标签体系**: 建立标准化的情绪标签体系
- **评分校准**: 定期校准评分标准，确保一致性
- **案例库建设**: 积累典型情绪分析案例，用于模型训练

## 九、监控与告警

### 9.1 监控指标

| 指标 | 说明 | 告警阈值 |
|------|------|----------|
| 情绪分析成功率 | 成功分析次数/总分析次数 | < 90% |
| 平均分析耗时 | 单次情绪分析平均耗时 | > 5s |
| 低分用户比例 | score < 0.4的用户占比 | > 20% |
| 高危用户数量 | score < 0.2的用户数 | > 5 |

### 9.2 告警策略

- **高危用户预警**: 检测到score < 0.2时触发高危预警
- **情绪恶化趋势**: 连续多次分析score持续下降时触发预警
- **异常波动检测**: 短时间内情绪分数剧烈波动时触发预警

## 十、合规与伦理

### 10.1 隐私保护

- **数据最小化**: 仅收集必要的情绪分析数据
- **用户知情权**: 明确告知用户情绪分析的目的和用途
- **数据删除权**: 支持用户删除情绪分析记录

### 10.2 伦理准则

- **非诊断性**: 情绪分析结果不作为医学诊断依据
- **专业转介**: 检测到严重情绪问题时建议寻求专业帮助
- **避免标签化**: 不对用户进行固定情绪标签

### 10.3 使用限制

- 本系统仅用于辅助情绪识别，不替代专业心理咨询
- 情绪分析结果仅供参考，不作为决策唯一依据
- 遇到危机情况应立即建议用户寻求专业医疗干预

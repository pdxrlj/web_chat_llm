# 记忆管理系统备案文档

## 一、系统概述

### 1.1 系统名称
智能记忆管理系统（Intelligent Memory Management System）

### 1.2 系统定位
基于向量数据库与大语言模型的智能记忆管理系统，用于在对话场景中持久化、检索和管理用户记忆数据，支持情感分析、冲突检测与智能合并。

### 1.3 技术架构
- **开发语言**: Python 3.12+
- **向量存储**: Milvus
- **嵌入模型**: Sentence Transformers（本地部署，CUDA加速）
- **重排序模型**: BGE-Reranker（本地部署，CUDA加速）
- **LLM集成**: OpenAI兼容API（用于记忆提取与冲突决策）
- **异步框架**: asyncio + AsyncOpenAI

## 二、核心模块说明

### 2.1 记忆数据模型

#### 2.1.1 记忆类型分类

| 类型 | 标识 | 说明 | 示例 |
|------|------|------|------|
| 事实性记忆 | fact | 用户的基本事实信息 | 姓名、职业、年龄 |
| 偏好性记忆 | preference | 用户的喜好与倾向 | 喜欢吃苹果、不喜欢颜文字 |
| 事件性记忆 | event | 用户发生的重要事件 | 搬家、换工作 |
| 上下文记忆 | context | 对话背景与环境信息 | 当前情境、环境信息 |

#### 2.1.2 记忆项数据结构

```python
@dataclass
class MemoryItem:
    id: str                              # 唯一标识符
    content: str                         # 记忆内容（简洁事实陈述）
    embedding: Optional[List[float]]     # 向量嵌入（用于语义检索）
    memory_type: MemoryType              # 记忆类型
    
    # 元数据
    user_id: Optional[str]               # 用户标识
    session_id: Optional[str]            # 会话标识
    created_at: datetime                 # 创建时间
    updated_at: datetime                 # 更新时间
    
    # 语义信息
    entities: List[str]                  # 提取的实体
    keywords: List[str]                  # 关键词
    
    # 冲突处理
    version: int                         # 版本号
    is_active: bool                      # 是否激活
    superseded_by: Optional[str]         # 被替代指向
    
    metadata: Dict[str, Any]             # 自定义元数据
```

### 2.2 记忆生命周期管理

#### 2.2.1 记忆提取流程

```
用户对话 → LLM分析提取 → 记忆项生成 → 向量嵌入 → 冲突检测 → 存储入库
```

**提取规则**:
1. 仅从 `role="user"` 的消息中提取用户偏好、事实、事件
2. 过滤无关寒暄和礼貌用语
3. 保留情感和态度词汇（喜欢/不喜欢/爱/讨厌等）
4. 记忆内容简洁明确，不添加前缀或标签

#### 2.2.2 向量嵌入生成

- **模型**: Sentence Transformers（本地加载）
- **设备**: CUDA GPU（优先）/ CPU（降级）
- **缓存**: `./models` 目录本地缓存
- **异步处理**: 通过 `asyncio.run_in_executor` 避免阻塞

#### 2.2.3 冲突检测机制

**检测策略**:
1. 向量相似度搜索（top_k条，宽松阈值初筛）
2. LLM智能决策（唯一决策者）
3. 字符重叠辅助判断（至少共享3个汉字）

**冲突类型**:
- `contradiction`: 矛盾冲突（新记忆与旧记忆相反）
- `update`: 更新关系（新记忆是旧记忆的补充或更新）
- `duplicate`: 重复记忆（内容高度相似）

### 2.3 冲突解决策略

| 策略 | 标识 | 说明 | 适用场景 |
|------|------|------|----------|
| 替换 | replace | 新记忆完全替换旧记忆 | 用户偏好发生改变 |
| 合并 | merge | 智能合并新旧记忆 | 信息互补不矛盾 |
| 保留旧 | keep_old | 保留旧记忆，忽略新记忆 | 新记忆价值较低 |
| 保留两者 | keep_both | 新旧记忆同时保留 | 信息不冲突且独立 |

**LLM决策流程**:
```
新记忆 + 相似记忆列表 → IntelligentDataProcessor → LLM评估 → 处理动作 + 合并内容 + 删除列表
```

## 三、存储层设计

### 3.1 向量存储（Milvus）

- **集合名称**: 可配置
- **向量维度**: 与嵌入模型输出维度一致
- **索引类型**: IVF_FLAT / HNSW（可配置）
- **度量类型**: L2 / IP（内积）

### 3.2 关系存储（PostgreSQL）

- **情感分析记录表**: `emotion_speculate`
- **字段**: id, session_id, role, query, emotion(JSON), created_at, updated_at

## 四、API接口说明

### 4.1 记忆管理接口

| 接口 | 方法 | 路径 | 说明 |
|------|------|------|------|
| 添加记忆 | POST | `/api/memory/add` | 从对话中提取并添加记忆 |
| 搜索记忆 | POST | `/api/memory/search` | 语义搜索相关记忆 |
| 获取全部记忆 | GET | `/api/memory/all` | 获取用户所有记忆 |
| 删除记忆 | DELETE | `/api/memory/delete` | 删除指定记忆 |
| 记忆报告 | POST | `/api/memory/report` | 生成用户画像分析报告 |

### 4.2 记忆报告生成

**输入**: session_id  
**处理流程**:
1. 根据session_id获取用户信息
2. 检索用户所有记忆数据
3. LLM分析生成结构化报告

**输出格式**:
```json
{
  "basic_info": ["基础信息要点"],
  "attention_needed": ["需关注内容（含风险标注）"],
  "current_status": ["当前生理心理状态"],
  "recent_events": ["近期重要事件"],
  "preferences": ["用户偏好"],
  "current_demands": ["当前需求"]
}
```

## 五、性能与安全

### 5.1 性能优化

- **异步架构**: 全链路异步IO，支持高并发
- **GPU加速**: Embedding和Reranker模型CUDA加速
- **本地模型**: 避免网络延迟，支持离线运行
- **批量处理**: 支持批量记忆提取和嵌入生成

### 5.2 数据安全

- **用户隔离**: 通过user_id实现数据隔离
- **会话隔离**: 通过session_id实现会话级数据管理
- **版本控制**: 记忆版本追踪，支持回溯
- **软删除**: 通过is_active标记，不物理删除数据

### 5.3 容错机制

- **模型加载失败降级**: GPU不可用时自动降级到CPU
- **LLM调用失败处理**: 异常捕获与日志记录
- **向量生成失败处理**: 跳过embedding直接存储
